# Research: Architecture Map of the To-Do Web Application

**Date**: 2026-06-16
**Researcher**: raalbue
**Repository**: todo_list (local, not under git)

## Research Question
The user has a "Python based clean architecture CRUD web application" they want to
enhance and improve. This document maps the existing system as-is — what exists,
where it lives, and how the pieces connect — to serve as a foundation for that later
work. (No recommendations are made here; this is documentation only.)

## Summary
The project is a **Django 6.0.5 web application** named `todo_project` containing a
single app, `todo_app`. It implements CRUD over three user-facing entities —
**ToDoList**, **ToDoItem**, and **Person** — plus a fourth model, **Account**, that
is defined but not wired into any view, URL, or admin registration.

Architecturally it follows Django's standard **MVT (Model–View–Template)** pattern
using **class-based generic views** (`ListView`, `DetailView`, `CreateView`,
`UpdateView`, `DeleteView`). There is no separate domain/use-case/repository layering
of the kind associated with "clean architecture" in the Hexagonal/Onion sense — the
business entities, persistence, and presentation are organized along Django's
conventional app layout. The data store is **PostgreSQL** (configured directly in
`settings.py`), though a SQLite `db.sqlite3` file also exists in the repo root.

## Detailed Findings

### Project / configuration layer (`todo_project/`)
- `todo_project/settings.py:33-41` — `INSTALLED_APPS` includes the default Django
  contrib apps plus `todo_app`.
- `todo_project/settings.py:83-92` — Active database is **PostgreSQL** (`todo_db` on
  `localhost:5432`, user `postgres`). The credentials and `SECRET_KEY`
  (`settings.py:23`) are hardcoded, and `DEBUG = True` (`settings.py:26`).
- `todo_project/settings.py:76-81` — The original SQLite config is commented out.
- `todo_project/settings.py:55-68` — Templates use `APP_DIRS: True` with an empty
  global `DIRS`, so templates are resolved from `todo_app/templates/`.
- `todo_project/urls.py:20-23` — Root URLconf mounts the admin at `admin/` and
  includes `todo_app.urls` at the site root.
- `manage.py`, `todo_project/wsgi.py`, `todo_project/asgi.py` — standard Django
  entry points generated by `startproject`.

### Data model layer (`todo_app/models.py`)
Four models, each with an explicit `db_table` via `Meta` (set in migration `0003`):
- **`ToDoList`** (`models.py:12-22`) — single field `title` (CharField, unique).
  `get_absolute_url` → `reverse("list", ...)`. Table: `todolist`.
- **`ToDoItem`** (`models.py:24-41`) — `title`, optional `description`,
  `created_date` (auto_now_add), `due_date` (defaults to one week out via the
  module-level helper `one_week_hence` at `models.py:9-10`), and a
  `ForeignKey` to `ToDoList` with `on_delete=CASCADE`. Default ordering by
  `due_date`. Table: `todoitem`.
- **`Account`** (`models.py:44-52`) — `email`, `password` (both CharField).
  **Not referenced by any view, URL, admin, or template** — defined only. Table:
  `account`.
- **`Person`** (`models.py:55-66`) — `email`, `address`. `get_absolute_url` →
  `reverse("person-detail", ...)`. Table: `person`.

Migrations: `0001_initial` (ToDoList/ToDoItem), `0002_account_person` (adds Account
and Person), `0003_rename_tables` uses `SeparateDatabaseAndState` to assign the
custom lowercase table names without touching the DB schema.

### View layer (`todo_app/views.py`)
All views are Django generic class-based views.

ToDoList / ToDoItem:
- `ListListView` (`views.py:14-16`) — lists all `ToDoList`s, renders
  `todo_app/index.html`.
- `ItemListView` (`views.py:18-28`) — items for one list, filtered by
  `list_id` from the URL; injects the parent `todo_list` into the context.
- `ListCreate` (`views.py:30-37`) — create a list (field: `title`); adds a
  `"title"` context label for the form heading.
- `ItemCreate` (`views.py:39-62`) — create an item; pre-populates the
  `todo_list` from `list_id` via `get_initial`, success redirects to that list.
- `ItemUpdate` (`views.py:64-80`) — edit an item; success redirects to the list.
- `ListDelete` (`views.py:83-87`) — delete a list, `success_url = reverse_lazy("index")`.
- `ItemDelete` (`views.py:89-98`) — delete an item, redirect back to its list.

Person:
- `PersonListView`, `PersonDetailView`, `PersonCreate`, `PersonUpdate`,
  `PersonDelete` (`views.py:101-136`) — full CRUD; create/update share
  `person_form.html`; delete redirects to `person-list`.

Note: `ItemCreate`, `ItemUpdate`, and `ItemDelete` rely on the **default Django
template names** (`todoitem_form.html`, `todoitem_confirm_delete.html`) rather than
explicit `template_name` attributes.

### URL layer (`todo_app/urls.py`)
- `urls.py:6-7` — `index` (`""`) and `list` (`list/<int:list_id>/`).
- `urls.py:9-12` — list create/delete.
- `urls.py:14-28` — item add / update (`item-update`) / delete, all nested under
  `list/<int:list_id>/`.
- `urls.py:30-34` — person list/detail/add/edit/delete under `persons/`.

### Template layer (`todo_app/templates/`)
- `base.html` — global layout; pulls in **Simple.css** from a CDN; clickable `<h1>`
  links home. All pages `{% extends "base.html" %}`.
- `todo_app/index.html` — renders `object_list` of lists; "Add a new list" button.
- `todo_app/todo_list.html` — items within a list, with add/delete-list buttons.
- `todo_app/todoitem_form.html` — shared create/edit form (`form.as_table`), with
  Submit/Cancel and a conditional "Delete this item" button.
- Person templates: `person_list.html`, `person_detail.html`, `person_form.html`,
  `person_confirm_delete.html`.
- Delete-confirmation templates: `todolist_confirm_delete.html`,
  `todoitem_comfirm_delete.html` (note the filename spelling), `todolist_form.html`.

### Admin (`todo_app/admin.py`)
- Registers only `ToDoItem` and `ToDoList` (`admin.py:4-5`). `Person` and `Account`
  are not registered.

### Tests (`todo_app/tests.py`)
- Empty placeholder — only the default `from django.test import TestCase` and a
  comment; no test cases defined.

## Architecture Documentation
- **Pattern**: Django MVT with class-based generic views; one app (`todo_app`).
- **Entity relationships**: `ToDoItem` *N→1* `ToDoList` (CASCADE delete). `Person`
  and `Account` are standalone with no relations to the to-do entities or to each
  other.
- **Routing convention**: items are nested under their parent list
  (`list/<list_id>/item/...`); persons live under a separate `persons/` namespace.
- **Naming**: explicit lowercase DB table names set via `Meta.db_table` and pinned by
  migration `0003`.
- **Presentation**: server-rendered Django templates; styling via the CDN-hosted
  Simple.css; navigation driven by inline `onclick="location.href=..."` handlers.
- **Persistence**: PostgreSQL configured in-code; a stray `db.sqlite3` remains in the
  repo from the earlier (commented-out) SQLite configuration.

## Observed Gaps in Wiring (factual, not recommendations)
- `Account` model: no view, URL, template, or admin registration references it.
- `Person` CRUD exists at the view/URL/template level but is **not linked from any
  navigation** (`base.html`/`index.html` only link to lists), so it is reachable only
  by typing `persons/` URLs directly.
- `Person` and `Account` are not registered in the Django admin.
- `tests.py` contains no tests.
